# 10. Settings

**AI Trading Assistant — Settings 페이지 설계 v1.0**
기준: 글로벌 디자인 시스템(00) 준수. Desktop First 1920×1080, 다크 테마 단일.

> **[제안 고지]** 원 스펙에는 Settings의 세부 항목이 정의되어 있지 않다. 본 문서의 항목 구성(표시 / 시계·시간대 / 보안 / 데이터 갱신 안내 / 관심 테마 / 초기화)과 페이지 전용 컴포넌트(토글·세그먼티드 컨트롤·라디오 프리뷰 카드·아코디언 헤더 등)는 00 문서의 원칙(P1~P5, §2~§11)에서 도출한 **제안(Proposal)** 이며, 확정 시 00 문서의 컴포넌트 절로 승격하는 것을 전제로 한다.

## 1. 전체 레이아웃 설명

### 1-1. 페이지 성격과 구조 판단

Settings는 데이터 소비 페이지가 아니라 **행동(설정 변경) 페이지**다. 따라서 다른 페이지의 "Hero 요약 + 드릴다운" 패턴 대신, 다음 4가지 원칙으로 구조를 단순화한다.

1. **저장 버튼 없음 — 즉시 적용(Instant Apply).** 모든 컨트롤은 조작 즉시 반영되고, 페이지 타이틀 우측 캡션과 해당 행의 "저장됨" 마이크로 피드백(`status.success`, 12px, 2초 후 소멸)으로 확인시킨다. 그 결과 이 화면에는 Primary Green 채움 CTA가 **0개**다(R3 상한 1개 이내 충족 — 유일한 Primary CTA는 비밀번호 모달 내부에만 존재).
2. **결과 미리보기 우선.** 등락 색상 모드처럼 "잘못 바꾸면 비싼" 설정은 텍스트 라디오가 아니라 **실제 등락 표기가 렌더링된 라디오 프리뷰 카드**로 제공한다. 사용자는 바꾸기 전에 결과를 본다.
3. **읽기 전용 정보는 같은 위계, 더 작은 면적으로.** 데이터 갱신 주기 안내(④)는 조작할 수 없지만, P3(신선도는 데이터의 일부다)에 따라 "이 제품의 숫자를 어떻게 읽어야 하는가"를 가르치는 카드로서 설정 스택과 같은 위계(카드 순서·앵커 내비 항목)를 유지한다. 다만 이 페이지는 스스로를 행동 페이지로 규정했으므로, 위계는 지키되 시각적 면적까지 정보 카드에 내줄 수는 없다 — ④는 기본 접힘(아코디언) 상태로 시작해 조작 가능한 카드보다 항상 작게 그려진다(구체 규칙은 §1-6).
4. **'정보 카드 ≤ 최대 조작 카드' 상한.** 이 페이지의 모든 읽기 전용 카드(현재 ④ 1개)는 접힘 기본 상태의 높이가 페이지 내 가장 큰 조작 가능 카드의 높이를 넘지 않는다. 카드 크기의 위계만으로도 "이 페이지가 무엇을 하는 곳인지"가 매 순간 증명되도록 하기 위함이다.

### 1-3. 콘텐츠 영역 2단 구성

- **좌: 앵커 내비게이션 — 3col(약 384px), 스티키.** 6개 섹션으로 점프. 항목 높이 36px, 라벨 13px/500 `text.secondary`, radius 8px. 호버 시 배경 `surface.card-hover` + 라벨 `text.primary`. 현재 스크롤 위치의 섹션은 좌측 2×16px 바 `brand.primary` + 라벨 `text.primary` 600(스크롤 스파이). 클릭 시 해당 카드로 스무스 스크롤(`motion.base` 200ms).
- **우: 설정 카드 스택 — 8col(약 1064px), 세로 1열, 카드 간 24px.** 우측 1col은 의도적 여백 — 설정 문장·컨트롤의 가독폭을 1064px로 제한해 시선이 좌(라벨)↔우(컨트롤) 왕복만 하게 한다.
- 카드는 전부 **Standard 등급 변형**(배경 `surface.card`, 보더 `border.default` 1px, radius 12px, 패딩 20px). Settings 카드는 숫자 카드가 아니므로 최소 높이 220px 규칙 대신 내용 기반 가변 높이를 적용한다(제안).

### 1-4. 설정 행(Setting Row) 공통 아나토미 — 제안

```
│  라벨 14px/500 text.primary                                    [컨트롤: 우측 정렬]  │  행 높이 56px
│  설명 12px/400 text.secondary (라벨과 4px 간격)                                     │  (설명 포함 시 68px)
├──────────────────────────────── border.subtle 1px ─────────────────────────────────┤
```

- 라벨 좌측 정렬 / 컨트롤 우측 정렬의 단일 축. 행 전체가 클릭 대상인 경우(토글 행)만 호버 배경 `surface.card-hover`.
- 컨트롤 히트 영역 최소 32×32px(§11-3).
- ④ 데이터 갱신 안내의 아코디언 헤더는 값을 바꾸는 컨트롤이 아니므로 이 아나토미를 따르지 않는다 — 별도 컴포넌트 규격은 §1-6 참조.

### 1-5. 페이지 전용 컨트롤 3종 — 제안

| 컴포넌트 | 스펙 |
|---|---|
| **토글(Switch)** | 트랙 40×24px, `radius.badge`(999px). OFF: 배경 `surface.sunken` + 보더 `border.default`, 노브 20px `text.secondary`. ON: 배경 `brand.primary`, 노브 `text.primary`(#F9FAFB). 전환 `motion.fast` 120ms. ON/OFF 텍스트 라벨을 토글 좌측 8px에 병기(색 단독 전달 금지, §11-2) |
| **세그먼티드 컨트롤** | 높이 36px, 컨테이너 배경 `surface.sunken` + 보더 `border.default`, radius 8px, 세그먼트 패딩 좌우 16px, 라벨 13px/500. 선택 세그먼트: 배경 `surface.selected` + 라벨 `text.primary` 600. 비선택: `text.secondary` |
| **라디오 프리뷰 카드** | 500×96px, radius 8px(`radius.control`), 배경 `surface.sunken`, 보더 `border.default` 1px. 선택 시: 보더 `brand.primary` 1px + 배경 `surface.selected` + 우상단 체크 아이콘 16px `brand.primary`. 호버(비선택): 보더 `border.strong` |

### 1-6. 읽기 전용 카드의 기본 접힘 규칙 — 제안 (리뷰 반영)

데이터 갱신 안내(④)처럼 조작 불가능한 정보 카드가 페이지에서 가장 큰 시각적 비중을 차지하면, "결론(P5) = 지금 내 표시 모드가 무엇인가"를 3초 안에 전달해야 하는 행동 페이지의 정체성과 충돌한다. 이를 컴포넌트 계약으로 구조화한다.

- **기본 상태 = 접힘(Collapsed).** 최초 진입·새 세션에서 ④는 항상 접힘 상태(1064×64px, 아코디언 헤더 1행)로 렌더링된다.
- **아코디언 헤더 컴포넌트.** 높이 64px, 카드 전체 폭(1064px)이 클릭 대상. 좌측 쉐브론 아이콘 16px `text.secondary`(접힘 `▸` / 펼침 `▾`, 회전 `motion.fast` 120ms) + 카드 제목 14px/600 `text.primary` + 우측 상태 요약 캡션 12px. 전 소스가 FRESH면 `text.tertiary`로 "전체 소스 정상 · 데이터 생성 07/10 07:12 KST"(00 §6-5와 동일한 타임스탬프 형식), 하나라도 DELAYED/STALE이면 `accent.yellow` 4px 점 + "N건 지연/오래됨"으로 대체 — 접힌 채로도 이상 신호는 새어나오게 해 P3를 완전히 가리지 않는다.
- **펼침 전환.** 헤더 클릭 시 카드가 1064×392px로 펼쳐지며 표·범례 전체가 나타난다(§2-2 참조). 높이 전환은 `motion.base` 200ms, 콘텐츠는 이미 로드되어 있으므로 별도 스켈레톤 없이 즉시 표시된다.
- **세션 내 상태 유지.** 사용자가 펼친 상태는 같은 브라우저 세션(sessionStorage) 동안 유지된다. 새 세션(재방문)에서는 항상 접힘으로 리셋 — "이 페이지는 매번 행동부터 보여준다"는 원칙을 지킨다.
- **딥링크 예외.** 다른 페이지의 [지연] 배지 툴팁이나 앵커 딥링크(§4 시나리오 D)로 진입한 경우, 접힘 기본값을 무시하고 자동으로 펼쳐진 채 스크롤 도착한다 — 정보를 찾아온 사용자에게 다시 클릭을 요구하지 않는다.
- **크기 상한.** 접힘 높이(64px)는 이 페이지의 가장 큰 조작 가능 카드(① 표시, 344px)보다 항상 작다(§1-1 원칙 4). 향후 정보성 카드가 추가되어도(§7 '알림 설정 카드' 등 읽기 위주 확장 포함) 이 상한을 상속한다.

### 3-3. 배치 순서의 논리

**사용 빈도 내림차순 + 파괴성 오름차순.** 매일 만질 수 있는 표시·시계가 최상단, 가끔 만지는 보안·테마가 중간, 읽기만 하는 데이터 안내가 그 사이(신선도 배지를 처음 본 사용자의 유입 지점이므로 보안 바로 아래), 파괴적 액션(초기화)은 항상 최하단 격리. 스크롤 0 상태에서 ①②전체 + ③상단이 보인다 — "설정 페이지의 결론 = 지금 내 표시 모드가 무엇인가"를 3초 안에 파악(P5). ④는 배치 순서상 이 자리(보안 바로 아래)를 유지하지만 §1-6 규칙에 따라 기본 접힘(64px)으로 시작하므로, 펼침 여부와 무관하게 페이지에서 시각적으로 가장 큰 카드는 항상 ①(조작 가능한 표시 카드, 344px)이다 — "이 페이지는 행동 페이지"라는 선언(§1-1)이 카드 크기 위계에서도 어긋나지 않는다.

## 4. 사용자 동선

### 시나리오 A. 미국 주식 위주 사용자가 등락 색을 글로벌로 전환 (첫 주 사용, 소요 약 10초)

- **경로 1 (Settings 경유, 2클릭)**: 사이드바 하단 Settings(1클릭) → 첫 카드 ①의 "글로벌 모드" 프리뷰 카드 클릭(2클릭) → 즉시 적용 + "차트 캔들 색도 함께 전환" 확인 캡션.
- **경로 2 (헤더 지름길, 2클릭)**: 헤더 아바타(1클릭) → 드롭다운의 등락 색상 모드 토글(2클릭). §7-4에 이미 정의된 지름길로, Settings는 이 지름길의 "정식 주소" 역할.
- **시선 흐름**: 좌상 타이틀 → ① 카드의 두 프리뷰(빨/파 vs 초록/빨강 실물 비교) → 선택. 텍스트를 읽지 않아도 색 표본만으로 결정 가능.

### 시나리오 B. 카페에서 자산 금액 가리기 (장중, 소요 약 5초)

- Settings(1클릭) → ① 카드 "금액 가리기 기본값" 토글(2클릭). **총 2클릭.**
- 급할 때는 어느 페이지에서든 `Shift+H`(제안 단축키)로 즉시 마스킹/해제 — Settings의 토글은 "다음 접속 시의 기본값"을 결정한다는 점을 행 설명 캡션이 명시.

### 시나리오 C. Asset 비밀번호 변경 (월 1회 미만, 소요 약 30초)

- Settings(1클릭) → 앵커 내비 "보안"(2클릭, 스크롤 점프) → [비밀번호 변경](3클릭) → Level 3 모달에서 입력·완료. **모달 도달까지 3클릭** — P5 상한 내.
- 실패 시 필드 보더 `status.danger` + 헬퍼 텍스트 교체(흔들림 애니메이션 없음, §6-3 준수). 성공 시 모달 닫힘 + 행에 `status.success` "변경됨" 캡션.

### 시나리오 D. "이 숫자에 왜 '지연' 배지가?" (임의 페이지에서 유입)

- 아무 카드의 [지연] 배지 호버 → 툴팁(마지막 갱신 시각 + "갱신 주기 안내" 링크, 0클릭) → 링크 클릭(1클릭) → Settings ④ 카드로 앵커 딥링크 진입, 이때 ④는 기본 접힘을 무시하고 자동으로 펼쳐진 채 도착한다(§1-6 딥링크 예외 규칙). **총 1클릭**으로 신선도 판정 기준·다음 갱신 예상 시각까지 도달. 사이드바 하단 데이터 생성 캡션(00 §6-5) 클릭으로도 동일 지점 도달.

### 시나리오 E. 저녁 복기 중 관심 테마 추가 (소요 약 15초)

- News 페이지 테마 필터 끝의 "테마 관리" 링크(1클릭) → Settings ⑤ 카드 앵커 진입 → 입력 필드에 "로봇" 타이핑 후 Enter(2번째 상호작용) → 칩 즉시 추가 + "5/10 → 6/10" 카운터 갱신. 또는 추천 칩 "( 로봇 + )" 클릭 1회로 완료.

### 키보드 동선

`Tab` 순서: 스킵 링크 → 사이드바 → 헤더 → 앵커 내비(↑↓ 탐색) → 카드 내 컨트롤 좌상→우하. 세그먼티드 컨트롤은 `←→`로 세그먼트 이동, 토글은 `Space`, 아코디언 헤더는 `Enter`/`Space`로 펼침·접힘 토글. 모든 포커스에 `focus.ring`(2px #34D399, offset 2px).

## 5. 디자인 의도

1. **"설정은 계약서가 아니라 스위치판이다."** 저장/취소 버튼 체계를 버리고 즉시 적용 + 행 단위 피드백을 택했다. 설정 항목이 10개 미만인 이 제품에서 폼 제출 모델은 과잉이며, Linear식 즉시성이 "절제와 정밀함"이라는 제품 성격에 부합한다. 부작용(실수로 변경)은 모든 컨트롤이 1클릭으로 원복 가능하고 파괴적 항목(초기화·비밀번호)만 모달 확인을 거치는 것으로 상쇄한다.
2. **가장 비싼 설정에 가장 큰 면적.** 등락 색상 모드는 이 제품에서 오독 비용이 가장 큰 설정이므로(00 §2-5의 결정 배경), 텍스트 한 줄짜리 토글이 아니라 500×96px 프리뷰 카드 2장 — 페이지에서 가장 큰 컨트롤 — 을 배정했다. "숫자가 주인공"(P1)의 설정 페이지식 번역: **결과 표본이 주인공**이다. ④가 §1-6 규칙으로 기본 접힘 상태를 갖는 것과 맞물려, 이 프리뷰 카드 2장을 품은 ① 카드는 페이지 전체에서도 항상 가장 큰 카드로 남는다.
3. **신선도 교육은 상주하되, 존재감은 접어둔다.** ④ 카드는 조작 불가한 안내지만, 정적 배치 생성이라는 이 제품의 데이터 특성(00 §12)을 사용자 멘탈 모델에 심는 유일한 상설 지면이므로 삭제하지 않는다. 그러나 이 페이지가 스스로 규정한 '행동 페이지' 정체성(§1-1)과 충돌하지 않도록, 카드 자체는 §1-6 규칙에 따라 기본 접힘 64px로 시작한다 — 존재(위계)는 지키되 면적(시각적 우선순위)은 조작 가능한 카드에 양보하는 절충이다. 배지 툴팁 → ④ 카드 딥링크는 접힘을 우회해 자동 펼침으로 도착시켜, "가르치는 지면"으로서의 기능은 필요한 순간에 100% 보존한다.
4. **잠금의 정직함.** 보안 카드 최상단의 `status.info` 고지는 §12가 의무화한 문구다. 과장된 보안 연출(방패 일러스트 등) 대신 정확한 보호 수준을 먼저 말하고, 그 아래에 실제 동작(변경·유휴 잠금·즉시 잠금)을 배치해 신뢰를 디자인한다.
5. **앵커 내비는 사이드바의 축소 재귀다.** 좌측 3col 앵커는 사이드바와 동일한 문법(좌측 바 인디케이터, 호버 배경)을 재사용해 학습 비용 0으로 페이지 내 이동을 제공한다. 6개 항목뿐이라도 ④가 펼침 상태에서는 표 때문에 길어지므로, 스티키 앵커가 "3클릭 도달"을 스크롤 위치와 무관하게 보장한다.

## 7. 개선 포인트

1. **설정 내보내기/가져오기 (v1.1 제안)**: 정적 환경 특성상 설정은 브라우저 로컬 저장 전제이므로 기기 간 자동 동기화가 없다. ⑥ 카드에 "설정 파일로 내보내기(JSON) / 가져오기"를 추가해 PC↔노트북 이전을 수동으로나마 지원. 비밀번호 해시 포함 여부는 가져오기 시 선택.
2. **앵커 딥링크 URL 규격화**: `#display` `#clock` `#security` `#data` `#themes` `#reset` 해시를 공식 규격으로 정의해, 다른 페이지의 툴팁·에러 배지·온보딩 코치마크가 Settings의 정확한 행으로 연결되게 한다(시나리오 D의 확장). `#data`로 진입 시 §1-6 딥링크 예외에 따라 ④가 자동 펼침되는 것도 이 규격에 포함한다.
3. **커맨드 팔레트 통합**: `Ctrl+K`에서 "글로벌 모드", "금액 가리기" 등 설정 항목을 직접 검색·토글(페이지 그룹의 하위로 '설정 액션' 그룹 추가). 자주 쓰는 2~3개 설정은 0클릭 이동으로 단축.
4. **알림 설정 카드 (향후)**: 브라우저 Notification 기반 "모닝리포트 생성 완료 알림", "관심 테마 뉴스 도착 알림"을 ④와 ⑤ 사이에 신설. 정적 환경에서는 열람 중 폴링 기반이라는 한계 고지 문구 필요. 이 카드 역시 읽기 위주 정보가 섞이므로 §1-6의 '정보 카드 ≤ 최대 조작 카드' 상한을 상속해 필요 시 접힘 기본값을 함께 검토한다.
5. **비밀번호 UX 보강**: 강도 미터(4단계, 회색→`status.success` 단색 전환), "비밀번호를 잊었다면 — 초기화 시 잠금 데이터 열람 불가" 안내 링크. 정적 환경에서 복구 수단이 없음을 설정 시점에 미리 고지하는 것이 사고 예방의 핵심.
6. **관심 테마 고도화**: 칩 드래그 정렬(우선순위가 News 정렬 가중치에 반영), 테마별 미니 스파크라인(최근 7일 관련 뉴스 건수) — 단, 색 절제 원칙상 스파크라인은 `text.secondary` 단색.
7. **모션 줄이기의 시스템 승격**: ①의 모션 옵션이 사용자 검증을 통과하면 00 문서 §10-2의 OS 설정 연동 규칙에 "사용자 수동 오버라이드" 조항으로 편입한다 — 페이지 제안이 시스템 규칙으로 흡수되는 개정 경로(00 문서 말미 원칙)를 따른다. §1-6의 접힘 규칙도 검증 후 동일한 경로로 00 문서에 전역 컴포넌트 계약으로 승격할 것을 함께 제안한다.
8. **1280px 축소 대응 상세화**: 사이드바 자동 접힘 시 앵커 내비를 상단 가로 탭바(높이 44px, 스크롤 스파이 유지)로 전환하고 카드 스택을 12col 전폭으로 확장하는 변형을 차기 버전에서 와이어프레임으로 확정할 것.
9. **Dashboard 연동 제안 — ⑤ 관심 테마 → 핵심 뉴스 우선 정렬.** 현재 ⑤ 카드 설명은 "News 페이지의 키워드·테마 필터 우선순위에 사용됩니다"로 한정한다 — 01(Dashboard) 문서에 실제로 존재하지 않는 기능("내 테마 뉴스" 정렬)을 임의로 언급하지 않기 위한 정정이다. 다만 Dashboard ④ 핵심 뉴스 카드(01 §3-5)에 "내 테마 우선 정렬" 필터 칩을 신설하면 이 관심 테마 목록을 그대로 재사용할 수 있다 — v1.1에서 01 문서 쪽 변경으로 제안해 둔다.
10. **전역 컴포넌트 제안 — 계좌/자산 배분 스택 바 색 규칙 통일.** Asset(08 §3-5)은 회색 3단계 교차, Portfolio(09 §3-1 Hero·§3-5)는 5색 고정 팔레트로 "계좌별·보유자산 비중 스택 바"라는 동일 위젯을 서로 다르게 그린다. 두 페이지 모두 같은 비밀번호 게이트 뒤에 있고 같은 성격의 정보(비중)를 다루므로, 색 처리는 페이지별 색 예산 소진 여부가 아니라 위젯 자체의 계약으로 고정되어야 한다. Settings 문서가 직접 다루는 범위는 아니지만, 00 문서에 `AccountAllocationBar` 같은 전역 컴포넌트 계약을 추가해 색 처리(예: 항상 회색 계열 + 직접 라벨, 또는 항상 R5 5색 팔레트)를 한 곳에서 고정하고 Asset·Portfolio가 이를 상속하도록 다음 개정에서 정리할 것을 제안한다.
11. **violet 인사이트 모티프 축소 제안.** Dashboard Hero, Macro Hero, News L3, FS 진단요약, Asset 목표카드·통화노출 등 5개 이상 페이지에서 동일한 형태(좌측 3px violet 바 + 회색 텍스트)가 반복되어 "시스템이 짚어준 특별한 판단"이라는 신호력이 옅어지고 있다는 지적이 이번 리뷰에서 함께 나왔다. Settings 문서 자체는 violet 모티프를 사용하지 않지만, 00 문서의 해당 컴포넌트 계약을 개정할 때 페이지별 최소 변주(아이콘 유무·위치·폭) 또는 Dashboard·FS 등 1~2개 핵심 페이지로의 용도 제한을 함께 검토할 것을 기록해 둔다.
12. **잠금 게이트 스크림 최적화 제안.** Asset·Portfolio의 비밀번호 게이트는 배경 마스킹 스켈레톤 위에 `overlay.scrim` 60% 검정을 겹치는데, 60% 불투명도에서는 아래 스켈레톤 골격이 거의 지각되지 않아 "해제 후 점프 없음"을 위한 사전 렌더링의 실효성이 낮다는 지적이 있었다. Settings ③ 카드의 "지금 잠그기" 버튼이 이 화면 전환을 직접 트리거하는 접점이므로, 스크림 불투명도를 40% 내외로 낮추거나 스켈레톤을 생략하고 페이드+상승 애니메이션만으로 전환하는 대안을 Asset·Portfolio 문서 쪽에서 검토할 것을 함께 제안한다.
