# 05. Stock Hub

**AI Trading Assistant — Stock Hub(종목 허브 패널) 설계 문서 v1.1**
적용 범위: Dashboard / News / Stock / Portfolio 등 **모든 페이지에서 종목을 클릭했을 때 열리는 전역 오버레이 컴포넌트**. 본 문서의 모든 토큰명·수치는 「00. 디자인 시스템 (Global Foundation)」을 변경 없이 인용한다. 사이드바·헤더는 00 문서 §6·§7 스펙을 그대로 따르며, 여기서는 본 패널과의 상호작용 특이사항만 기술한다.

> **개정 노트 — v1.0 → v1.1 (디자인 리뷰 반영)**
> 사내 리뷰에서 지적된 6건 중, Stock Hub 패널 자체의 화면 요소에 해당하는 항목(①②③)은 아래 각 섹션 본문에 직접 반영되어 있다. 다른 페이지(00 Foundation·03 Asset·04 Stock·09 Portfolio)에 걸친 항목(④⑤⑥)은 이 패널에는 해당 요소가 없어 본문에 등장하지 않으므로, 시스템 일관성을 위해 필요한 연동 조치만 여기 요약해 둔다.
>
> - **①③ [critical ×2] 패널 표면 토큰 — "호출 지점마다 다르다"와 "Level 2인데 카드 밝기다"가 서로 반대 방향을 가리킴.** 04 §3-5와 09 §3-6 모두 독자적으로 `surface.elevated`를 명시해, 05 v1.0의 `surface.card`와 두 문서 모두에서 정면으로 어긋나 있었다(①) — 그리고 두 문서가 약속 없이도 똑같이 `elevated`를 택했다는 사실 자체가 "애초에 05의 선택(`surface.card`)이 Level 2 오버레이의 밝기 위계를 satisfy 못 한다"는 지적(③)이 근본 원인이었음을 뒷받침한다. v1.1은 `surface.card`도 `surface.elevated`도 아닌 **제3의 전용 토큰 `surface.overlay-panel`**을 신설해 두 지적을 동시에 해소한다. 상세 근거는 §1-3 「표면 결정 노트」, 반영 지점은 §1-3·§3-2·§6-2. **04 §3-5, 09 §3-6, 00 §4-3도 이 토큰으로 갱신되어야 한다** — 05가 이 컴포넌트의 단일 진실 소스이므로 두 호출 문서는 후속 동기화 커밋에서 기존 `surface.elevated` 표기를 교체한다(본 파일 범위 밖이므로 여기서는 갱신하지 않음).
> - **② [critical] "다가오는 일정" 행 클릭이 죽은 링크였다.** 제품 IA(Dashboard/Macro/News/Stock/FS/TA/Asset/Portfolio/Settings + Stock Hub) 어디에도 "공시 원문·IR 페이지"라는 목적지가 없었다. v1.1은 일정 유형 5종(실적/공시/IR/SEC/배당) 각각에 목적지·이동 방식·아이콘을 명시하고, 목적지가 없는 유형(배당)은 클릭을 비활성화해 죽은 링크를 제거했다. §1-4, §2-2, §2-3, §3-2 D에 반영. 09 Portfolio의 `EventRow`는 이 컴포넌트를 그대로 재사용하므로 별도 정의 없이 동일하게 적용된다.
> - **④ [major] "계좌별 자산 배분" 위젯 색 처리 불일치(Asset 무채색 vs Portfolio 5색).** Stock Hub에는 이 위젯이 없어 본문에 등장하지 않는다. 권장 해소안: 계좌/자산 배분 스택 바를 전역 컴포넌트로 승격하고 **항상 무채색 회색 계열 + 직접 라벨**로 고정한다(임의 카테고리에 고정 팔레트를 배정하는 것은 §6-1의 "의미 색 4종 상한"이 지키는 원칙 — 색은 의미에만 쓴다 — 과 충돌하므로, 두 페이지 모두 무채색 규칙을 상속해야 한다). 00 문서에 컴포넌트 계약으로 신설하고 03 Asset·09 Portfolio가 상속하는 일은 본 파일 범위 밖이다.
> - **⑤ [major] `accent.violet` 해석 레이어 모티프가 5개 페이지·6곳 이상에서 반복.** Stock Hub는 v1.0부터 violet을 이미 배제하고 있었다(§6-1). 권장 해소안(Dashboard·FS 두 페이지로 용도를 좁히고 나머지는 일반 캡션으로 대체)은 본 패널의 기존 결정과 자연히 합치하므로, §6-1에 정합성 확인 문구만 보강했다.
> - **⑥ [minor] 잠금 게이트 스크림 60% 아래 스켈레톤이 거의 지각되지 않음.** Stock Hub는 잠금 게이트가 없어 해당 사항 없음. Asset·Portfolio에는 스크림을 40% 내외로 낮추거나 스켈레톤을 제거하고 페이드+상승 전환만으로 대체할 것을 권장 — 본 파일 범위 밖.

## 1. 전체 레이아웃 설명

### 1-1. 컴포넌트 정의

Stock Hub는 "종목 하나에 대한 **3초 요약 + 1클릭 분기**"를 담당하는 전역 패널이다. 페이지가 아니라 **오버레이 컴포넌트**이며, 다음 트리거에서 항상 동일하게 호출된다.

| 호출 지점 | 트리거 예시 |
|---|---|
| Dashboard | 관심종목 Row(52px) 클릭, Compact 지표 카드의 구성 종목 클릭 |
| News | 기사 본문·헤드라인에 붙은 티커 칩(예: `삼성전자 005930`) 클릭 |
| Stock | 종목 스크리너·리스트 행 클릭 |
| Portfolio / Asset | 보유종목 Row 클릭 |
| 헤더 글로벌 검색 | 검색 결과 '종목' 그룹에서 Enter (§7-2) |

패널의 역할은 명확히 제한된다: **① 종목 개요(시세·핵심 스탯) 즉시 표시 ② TradingView 차트로 분기 ③ Financial Statements로 분기 ④ 관련 일정·뉴스 열람 및 분기.** 심층 분석은 패널이 아니라 목적지 페이지가 담당한다 — 패널은 목적지가 아니라 분기점(Hub)이다.

### 1-2. 모달 vs 우측 슬라이드 패널 — ★결정: **우측 슬라이드 패널**

| 판단 기준 | 모달 (Level 3) | 우측 슬라이드 패널 | 판정 |
|---|---|---|---|
| 원본 문맥 유지 | 스크림으로 배경 차단 — 뉴스 리스트·관심종목이 가려짐 | 배경 페이지가 그대로 보이고 조작 가능 | **패널 승** |
| 연속 탐색(여러 종목 훑기) | 닫기 → 다시 열기 반복 강제 | 다른 종목 행 클릭 시 내용만 교체(패널 유지) | **패널 승** |
| 컴포넌트 성격과의 정합 | 모달은 "결정·확인"용 — 시스템 내 용례는 비밀번호 게이트(§6-3) | 참조·드릴다운용 보조 화면에 적합 | **패널 승** |
| 3클릭 도달(P5) | 닫기 동작이 클릭 예산을 소모 | 허브 → 목적지 직행, 복귀 경로가 시각적으로 살아 있음 | **패널 승** |
| 1920px 화면 활용 | 중앙 640~720px만 사용, 좌우 유휴 | 우측 480px 사용, 좌측 1,200px+ 문맥 보존 | **패널 승** |
| 강한 집중 유도 | 우세 | 열세 | 모달 승 — 그러나 본 컴포넌트에 불필요 |

**결정 근거 요약**: Stock Hub의 사용자는 "리스트를 훑다가 → 하나를 열어보고 → 다시 리스트로" 움직인다. 이 왕복 동선에서 배경을 죽이는 모달은 매 왕복마다 인지 비용과 클릭 비용을 부과한다. 반면 슬라이드 패널은 리스트(원인)와 패널(결과)을 한 화면에 공존시켜, 연속 스캐닝 시 **행 클릭 1회 = 종목 전환 1회**라는 최소 비용 구조를 만든다. 모달은 시스템 내에서 "되돌릴 수 없는 결정"(잠금 해제 등)에만 예약한다.

### 1-3. 패널 골격 치수

```
위치      : 우측 고정(right 0), 상단은 헤더 아래(top 64px)부터 뷰포트 하단까지
높이      : 1080 기준 1,016px (뷰포트 높이 − 64px)
폭        : 480px (1920·1440 공통) / 1280px에서 440px
패딩      : 좌우 24px (Hero 카드 패딩 기준)
표면      : surface.overlay-panel (#253040) — v1.1 신설 토큰, surface.card에서 전환(아래 표면 결정 노트 참고)
보더      : 좌측 1px border.strong (#4B5563)
그림자    : Level 2 스펙 — 0 8px 24px 검정 40% (오버레이 레이어이므로 허용, §4-3)
라운드    : 0 (화면 우측 가장자리에 밀착 — radius.card 미적용)
스크림    : 없음 (overlay.scrim은 Level 3 모달 전용 — 비모달 오버레이로 운용)
z-순서    : 페이지 콘텐츠 위, 드롭다운·툴팁(Level 2)·모달(Level 3) 아래
```

**표면 결정 노트 (v1.1 개정)**: v1.0에서는 480×1,016px 대면적에 `surface.elevated`를 칠하면 "일반 카드 배경으로 남용 금지" 조항의 취지(밝은 표면 희소성 유지)를 훼손한다고 보아, 표면을 `surface.card`로 유지하고 그림자 + `border.strong`만으로 층을 표현했다. 그러나 이 절충은 두 가지 문제를 낳았다.

1. **밝기 위계 붕괴.** 00 문서 §4-3 엘리베이션 표는 "같은 평면에서는 그림자를 쓰지 않는다. 깊이 = 표면 밝기 + 보더"라는 원칙 아래 Level 1(카드)과 Level 2(드롭다운·팝오버·모달)를 표면 밝기 단계로 명확히 분리한다(P4). 스스로 'Level 2 오버레이'라 정의한 이 패널이 Level 1과 동일한 `surface.card`를 쓰면, 정작 깊이 구분이 가장 필요한 순간 — 리스트 위에 480px짜리 패널이 떠 있는 순간 — 에 시각적 분리가 그림자와 좌측 보더 1px에만 의존하게 되어 다크 테마에서 약해진다.
2. **호출 문서 간 표기 분산.** 04 Stock §3-5, 09 Portfolio §3-6 모두 이 패널을 인용하면서 05 v1.0의 `surface.card`를 따르지 않고 각자 독자적으로 `surface.elevated`(#2A3648)를 명시해, 두 문서 모두 05와 어긋났다. 절충안 자체가 "Level 2인데 카드 밝기"라는 애매한 상태였기 때문에, 두 호출 문서가 약속 없이도 동일하게 "그래도 Level 2다운 밝기가 필요하다"는 방향으로 이탈한 것이다.

v1.1은 두 문제를 함께 푸는 **제3의 값**을 신설한다: **`surface.overlay-panel` (#253040)** — `surface.card` #1F2937과 `surface.elevated` #2A3648의 중간 명도. 480px급 대면적 비모달 오버레이는 전 종류가 이 토큰 하나만 쓴다.

- `surface.card`(일반 카드)보다 밝아 Level 2 오버레이로서 배경과 명확히 분리되고(P4 충족),
- `surface.elevated`(드롭다운·팝오버·모달처럼 작고 일시적인 표면)보다는 어두워 "밝은 표면은 희소하게" 원칙을 지키며, 패널 위에 뜨는 소형 오버레이(툴팁·팝오버)와도 밝기 위계가 겹치지 않는다.
- 패널 내부 Row 호버는 새 파생 토큰 `surface.overlay-panel-hover` (#283344, `surface.card`→`surface.card-hover`와 동일한 상승폭 규칙을 적용)를 쓴다. #253040(overlay-panel)과 #2A3648(elevated) 사이에 정확히 끼도록 R·G·B 세 채널 모두를 단조 증가시켜 잡은 값이므로, 호버가 팝오버보다 밝아지는 역전이 생기지 않는다. 패널 위에 뜨는 툴팁·팝오버는 규정대로 `surface.elevated`를 유지해, **배경(`surface.card`, 가장 어두움) → 패널(`surface.overlay-panel`, 그다음) → 패널 내 호버(`surface.overlay-panel-hover`, 그다음) → 팝오버(`surface.elevated`, 가장 밝음)**의 4단 밝기 위계가 성립한다.

**연동 필수 사항**: 04 Stock §3-5, 09 Portfolio §3-6의 패널 표면 스펙은 이번 개정으로 `surface.overlay-panel` (#253040) + 좌측 `border.strong` 1px + Level 2 그림자로 통일되어야 한다 — 04·09 모두 기존에 독자적으로 명시했던 `surface.elevated` 표기를 이 신규 토큰으로 교체하고, 내부 Row 호버가 있다면 `surface.overlay-panel-hover`(#283344)로 함께 맞춘다. 00 문서 §4-3 엘리베이션 표에도 "대면적 비모달 오버레이(Stock Hub 등)" 전용 행으로 이 토큰을 추가해야 한다. 05는 이 컴포넌트의 유일한 정의 출처(single source of truth)이므로, 위 세 문서의 실제 수정은 05 개정 확정 이후 별도 동기화 커밋에서 진행한다(본 파일은 05 자체만을 다룬다).

### 1-4. 열림 · 닫힘 · 교체 동작 (전역 계약)

| 동작 | 스펙 |
|---|---|
| 열림 | 우→좌 슬라이드 인: X +24px → 0 이동 + 불투명도 0→1, `motion.base` 200ms, 감속 곡선 |
| 닫힘 | 역방향 120ms (퇴장은 더 빠르게 — §10-2) |
| 닫기 트리거 | ✕ 버튼(28×28px, 히트 32px) / `Esc` / 배경 영역 클릭 / 사이드바로 페이지 전환 시 자동 닫힘 |
| 종목 교체 | 패널 열림 상태에서 다른 종목 트리거 클릭 → 패널 이동 없이 **내용만 크로스페이드 120ms 교체** |
| 동일 종목 재클릭 | 무동작(패널 유지) — 깜빡임 금지 |
| 배경 페이지 | 스크롤·클릭 모두 유효. 단, 배경 클릭 중 "종목 트리거"는 교체로, 그 외 영역은 닫기로 해석 |
| 목적지 이동 시 (v1.1: 유형 확장) | **패널 유지·외부 새 탭**: TradingView 차트, 일정 유형 [공시]·[IR]·[SEC](§3-2 D 참고) — 전부 `↗` 아이콘 병기. **패널 닫힘·내부 이동**: Financial Statements(재무제표 분석 버튼), 일정 유형 [실적](FS 페이지 실적 섹션으로 이동), 관련 뉴스 행(News 상세) |
| 페이지 전환 시 닫는 이유 | 새 페이지의 첫 화면 요약(P5)을 480px 패널이 가리지 않게 하기 위함 |
| 동작 줄이기(OS) | 슬라이드·크로스페이드 제거, 즉시 전환 (§10-2) |

### 1-5. 키보드 · 포커스 계약

| 키 | 동작 |
|---|---|
| `Esc` | 패널 닫기 (열린 툴팁·팝오버가 있으면 그것부터 역순 해제 — §11-3) |
| `Tab / Shift+Tab` | 패널 내부 순환(포커스 트랩): 닫기 → 즐겨찾기 → CTA 2개 → 일정 행(클릭 불가 행은 트랩에서 제외) → 뉴스 행 → 푸터 |
| `↑ ↓ + Enter` | 일정·뉴스 리스트 탐색 및 진입. 클릭 불가 일정 행(배당)은 `↑↓`로 지나가되 `Enter`는 무동작 |
| `T` / `F` | (패널 포커스 상태, 입력 필드 외) TradingView 열기 / Financial Statements 이동 |
| 열림 시 포커스 | 패널 헤더의 종목명 블록으로 이동, 스크린리더에 "삼성전자 종목 패널, 현재가 팔만사천삼백 원, 플러스 0.95퍼센트" 낭독 |
| 닫힘 시 포커스 | 호출한 트리거 요소로 복귀 |
| 포커스 링 | 전 요소 `focus.ring`(#34D399 2px, offset 2px), 키보드 포커스에만 표시 |

스크린리더용으로 등락은 기호가 아닌 "플러스/마이너스" 텍스트 레이블을 제공한다(§11-3). 클릭 불가 일정 행(배당)은 `aria-disabled="true"`와 함께 "정보 전용, 이동 없음"을 부가 낭독한다.

### 1-6. 사이드바 · 헤더 특이사항

- **사이드바 활성 메뉴는 변경하지 않는다.** 패널은 페이지 이동이 아니므로, Dashboard에서 열었으면 Dashboard가 활성 상태를 유지한다.
- **헤더 브레드크럼도 변경하지 않는다.** `Stock / 삼성전자 005930` 형식의 컨텍스트 갱신은 Financial Statements 등 실제 페이지 이동 시에만 발생한다.
- 헤더의 글로벌 검색·듀얼 시계는 패널 열림 중에도 완전히 사용 가능하다(패널이 top 64px 아래에 위치하는 이유). 검색에서 다른 종목 Enter 시 패널 내용 교체.

## 3. 카드 배치

패널은 12컬럼 페이지 그리드 **밖**의 오버레이이므로, 내부는 단일 컬럼(432px = 480 − 24×2) 수직 스택으로 배치한다. 호출 원(배경) 페이지의 그리드는 일절 변형하지 않는다.

### 3-1. 블록 총괄표

| 블록 | 순서 | 높이 | 내부 패딩 | 대표 타이포 | 정보 우선순위 |
|---|---|---|---|---|---|
| A. 패널 헤더 | 1 (sticky) | 72px (축약 56px) | 좌우 24px | `type.h2` 20px/600 | 1 — 무엇(종목 식별) |
| B. 시세 개요 | 2 | 184px | 상하 20px | `type.num-display` 32px | 1 — 얼마(현재가·등락), 2 — 언제(as-of·세션) |
| C. 액션 허브 | 3 | 92px | 상하 16px | 버튼 14px/600 | 3 — 어디로(분기) |
| D. 다가오는 일정 | 4 | 236px* | 행 좌우 16px | `type.body` 14px | 4 — 무슨 일이 예정 |
| E. 관련 뉴스 | 5 | 340px+ | 행 좌우 16px | `type.body` 14px | 5 — 무슨 일이 있었나 |

블록 간 구분은 `border.subtle` 1px 풀폭 디바이더. 블록 간 수직 리듬은 카드 간 거터와 동일한 24px 기준.
\* D 블록 푸터는 v1.1에서 링크(전체 일정 보기 →)에서 비클릭 캡션(초과 건수 안내)으로 변경되었다 — 제품 IA에 일정 전체를 모아 보는 목적지가 없어, 없는 페이지로 가는 링크를 다는 대신 "표시 개수가 전부"라는 사실만 안내한다(§3-2 D, §7-8).

### 3-2. 블록별 상세

**A. 패널 헤더 (480×72px, sticky)**
- 좌: 종목명 `type.h2` 20px/600 `text.primary` + 즐겨찾기 별 20px(간격 8px). 별 활성 시 `accent.yellow` 채움, 비활성 시 `text.tertiary` 아웃라인 — 토글로 관심종목 등록.
- 우: 티커·시장 캡션 `type.caption` 12px `text.secondary`(예: `005930 · KOSPI`) + 닫기 ✕ 28×28px(히트 32px, `text.secondary` → 호버 `text.primary`).
- 스크롤 임계(개요 블록이 화면 밖으로 나가는 시점)에서 §2-4의 컴팩트 모드로 전환.

**B. 시세 개요 (480×184px)**
- 1행: 현재가 `type.num-display` 32px/700 `text.primary` — 패널 내 유일한 display 타이포(P1). 우측에 등락 `type.num-md` 16px/500 `market.up`/`market.down`, ▲▼ + 부호 병행(R4), 진짜 마이너스(−) 사용.

## 5. 디자인 의도

- **허브는 얕게, 목적지는 깊게.** 패널에 차트 미리보기·재무 요약까지 욱여넣으라는 요구가 예상되지만 의도적으로 배제했다. 패널의 존재 이유는 "3초 판단 → 1클릭 분기"이며, 깊이는 TradingView와 Financial Statements가 담당한다. 정보를 늘릴수록 허브의 반응 속도(스캔 속도)가 죽는다 — 이것이 "많아 보이되 복잡해 보이지 않는" 해법이다(§0-1).
- **슬라이드 패널 = 문맥 보존 장치.** 투자 판단은 항상 "무엇을 보다가 이 종목에 왔는가"라는 문맥 위에서 이뤄진다. 뉴스 헤드라인 옆에서 시세를 보고, 관심종목 리스트 옆에서 개별 종목을 본다. 스크림 없는 비모달 설계는 이 병렬 대조를 구조적으로 보장한다.
- **숫자가 주인공(P1).** 패널에서 가장 큰 요소는 현재가 32px이며, 종목명(20px)보다 크다. 스탯 6종은 전부 tabular로 우측 정렬해 자릿수 비교가 스캔만으로 가능하다.
- **세션 인식은 계약이다(P3, §9-4).** 야간선물 오독 사고의 재발 방지를 위해, 패널의 시세 블록은 "유효 세션 밖 = CLOSED-SNAPSHOT 강제"를 컴포넌트 계약으로 내장한다. 어느 페이지에서 호출되든 이 계약은 동일하게 작동한다 — 전역 컴포넌트로 설계한 핵심 이유다.
- **전역 단일 컴포넌트 = 학습 비용 1회.** Dashboard·News·Portfolio 어디서 열어도 구조·단축키·닫기 동작이 완전히 동일하므로, 사용자는 패널 사용법을 한 번만 배운다. 페이지별 변형(추가 버튼, 다른 배치)은 금지한다.
- **표면 토큰도 시스템 계약이다(v1.1).** "이 정도 크기면 카드로 충분하다"는 절충은 편해 보이지만, 실제로는 밝기 위계라는 시스템 규칙에 예외를 하나 만드는 것과 같다. 예외 하나는 반드시 다음 호출 문서의 또 다른 해석(04는 카드, 09는 elevated)을 낳는다. `surface.overlay-panel`을 신설해 "대면적 비모달 오버레이는 이 값 하나"라는 규칙으로 되돌린 것은, 개별 화면의 판단을 줄이고 토큰이 판단을 대신하게 하는 것이 유지보수 비용을 낮춘다는 원칙의 재확인이다.
- **모든 클릭에는 목적지가 있어야 한다(v1.1).** "언젠가 만들 상세 페이지"를 가정한 링크는 그 페이지가 만들어지기 전까지는 사용자에게 실패 경험만 준다. 일정 행의 유형별 목적지 계약은 지금 실제로 존재하는 목적지(FS 페이지, 외부 공시·IR·SEC 사이트)만 클릭 가능하게 하고, 존재하지 않는 목적지(배당 상세)는 정보 표시로 격하해 죽은 링크 자체를 시스템에서 없앤다.

## 6. 색상 및 여백 사용 이유

### 6-1. 의미 색 4종 상한(R1) 검증

패널 열림 상태의 뷰포트에 존재하는 유채색은 정확히 4계열이다.

| 색 | 사용처 | 역할 |
|---|---|---|
| `market.up` / `market.down` (빨강/파랑, 한국 모드) | 등락 텍스트·▲▼, 배경 페이지의 등락 열 | 등락 방향 (2색 1계열로 계산) |
| `brand.primary` (초록) | TradingView CTA 1개, 사이드바 활성 인디케이터 | 핵심 행동 |
| `accent.yellow` | D-3 이내 임박 배지, 즐겨찾기 별, (해당 시) 지연·STALE 배지 | 주의·상태 |
| `badge.live` | 세션 점 8px (장중 한정) | 신선도 |

- 일정 유형 배지(실적/공시/배당/IR/SEC) 5종을 색으로 구분하고 싶은 유혹이 가장 큰 지점이나, **회색 아웃라인 + overline 타이포**로 통일했다(P2·R2). 유형에 색을 주는 순간 등락·상태 색과의 오독 경쟁이 시작되고 R1 상한이 깨진다. v1.1에서 추가한 외부 링크 아이콘(`↗`)과 배당 행의 톤다운(`text.tertiary`)도 전부 무채색이므로 이 4색 상한에 영향을 주지 않는다.
- `accent.violet`(해석 레이어 전용)은 이 패널에서 의도적으로 미사용 — 패널은 사실(시세·일정·뉴스)만 다루고 해석은 담지 않기 때문이다. 이번 리뷰에서 violet 해석 레이어 모티프의 전역 사용 범위 자체가 Dashboard·FS 두 페이지로 좁혀지는 방향으로 결정되었으므로(리뷰 항목 ⑤), 패널 단위에서 애초에 violet을 배제한 이 결정은 결과적으로 전사 방향과 정확히 합치한다 — 별도 수정 없이 그대로 유지한다.
- 재무제표 분석 버튼을 아웃라인으로 강등한 것은 R3(Primary 채움 1개) 준수이자, "차트 확인"이 이 사용자군의 최빈 행동이라는 우선순위 판단이다.

### 6-2. 표면·보더 운용

- 패널 표면 `surface.overlay-panel`(#253040) + 좌측 `border.strong` + Level 2 그림자 — v1.1에서 표면을 `surface.card`로부터 전환해, 다크 테마에서 층 분리를 밝기 자체가 우선 담당하고(P4) 보더·그림자는 보조 신호로 남도록 정정했다(§1-3 표면 결정 노트). 스크림을 쓰지 않는 대신 `border.strong`(일반 카드의 `border.default`보다 한 단계 강함)으로 경계를 또렷하게 만든다.
- 블록 구분은 배경색 차이가 아니라 `border.subtle` 디바이더 — 표면 단계를 아끼면 STALE 강등(`text.secondary`)·호버(`surface.overlay-panel-hover`) 같은 상태 표현의 대비 여유가 산다.
- 클릭 불가 행(배당)은 표면 자체는 바꾸지 않고 텍스트만 `text.tertiary`로 낮춘다 — 표면색까지 바꾸면 "다른 종류의 카드"로 오독될 위험이 있어, 상태 차이는 항상 텍스트 명도로만 표현한다는 기존 원칙(STALE 강등과 동일 패턴)을 그대로 따른다.

### 6-3. 여백 리듬 (4pt 스케일)

- **24px**: 패널 좌우 패딩(Hero급 — 패널은 이 화면의 주인공 표면) 및 블록 간 기준 리듬.
- **16px**: 액션 허브 상하, Row 좌우 패딩 — 밀도가 필요한 리스트 영역은 한 단계 조인다.
- **12px**: 버튼 간, 스탯 열 간, 배지·텍스트 간 — 형제 요소 간격.
- **8px**: 아이콘↔텍스트, 세션 점↔라벨.
- **4px**: 라벨↔값 수직 간격 — 라벨과 값을 한 덩어리로 읽히게 하는 최소 간격.

간격이 좁을수록 강한 소속, 넓을수록 강한 단절이라는 단일 규칙만 사용해, 480px의 좁은 폭 안에서 6개 블록이 보더 없이도 그룹으로 인지되게 한다.

## 7. 개선 포인트

1. **딥링크 상태**: URL 해시(예: `#stock=005930`)로 패널 열림 상태를 표현하면 새로고침·공유 시에도 동일 종목 패널이 복원된다. 정적 GitHub Pages 환경에서도 해시 라우팅은 서버 없이 동작하므로 구현 부담이 낮다.
2. **최근 본 종목 내비게이션**: 패널 헤더에 ← → 히스토리 버튼(28×28px)을 추가해 연속 스캐닝 중 직전 종목 복귀를 1클릭화. 최근 5종목은 글로벌 검색의 기본(빈 입력) 결과로도 재사용.
3. **TradingView 심볼 매핑 설정**: Settings에 "내 TradingView 레이아웃 URL 템플릿" 항목을 두어, 사용자별 차트 레이아웃·거래소 프리픽스(KRX:005930 / NASDAQ:NVDA)를 커스터마이즈.
4. **매매일지 연동 퀵 액션**: 패널 푸터에 "이 종목으로 일지 쓰기" 고스트 버튼을 추가하면 복기 동선(시나리오 C)이 1클릭 단축된다. 단 R3(채움 CTA 1개)을 지키기 위해 고스트 스타일 고정.
5. **정적 데이터 제약 수용**: 시세·일정·뉴스는 빌드 시점 스냅샷이므로, 신선도 판정 주기 T를 소스별로 분리 설정한다(시세 T=장중 20분/뉴스 T=6시간/일정 T=24시간). 데이터 파일에는 값과 함께 as-of·유효 세션 메타를 항상 포함시킨다(00 문서 §12 전제 동일).
6. **핀 고정(비교) 모드**: 패널 우상단에 핀 아이콘을 추가해 고정 시 배경 클릭으로 닫히지 않게 하고, 향후 2종목 나란히 비교(패널 폭 2×440px)로 확장 가능성을 열어둔다 — 단 1280px 이하에서는 비활성.
7. **모바일 대응(공식 범위 외)**: 1152px 미만·터치 환경에서는 우측 패널 대신 하단 풀스크린 시트로 전환하는 변형을 별도 문서로 정의할 것 — 열림/닫힘 계약(§1-4)과 블록 구조는 동일하게 유지한다.
8. **종목별 일정 캘린더 페이지(v1.1 신설 항목)**: 현재 제품 IA에는 "전체 일정"을 모아 보는 목적지가 없어, D 블록은 최대 4건만 표시하고 초과분은 링크가 아닌 카운트 캡션으로만 안내한다(§3-1 각주, §3-2 D). 향후 이런 목적지 페이지가 생기면 캡션을 다시 링크로 승격하고, 09 Portfolio의 `EventRow`도 동일하게 갱신한다.
