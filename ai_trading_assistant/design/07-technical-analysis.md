# 07. Technical Analysis

**AI Trading Assistant — Technical Analysis 페이지 설계 v1.0**
상태: **Coming Soon (모듈 슬롯 프리뷰 단계)**. 본 문서는 00. 디자인 시스템(Global Foundation v1.0)의 토큰·수치를 변경 없이 인용한다.
기준 해상도: 1920×1080, Desktop First, 다크 테마 단일.

## 1. 전체 레이아웃 설명

### 1-1. 페이지의 역할 정의

Technical Analysis는 아직 기능이 공개되지 않은 페이지다. 그러나 이 페이지의 목표는 "공사 중 안내판"이 아니라 **"미래 화면의 골격을 미리 입주시키는 것"**이다. 설계 핵심은 다음 세 가지다.

1. **레이아웃 연속성(Layout Continuity)**: 오늘 보이는 Coming Soon 화면의 그리드 골격은 향후 기능이 공개된 뒤의 실제 화면 골격과 동일하다. 각 미래 기능(지표 대시보드, 차트 패턴 인식, 백테스트)은 오늘부터 자기 자리(슬롯)를 점유하고 있으며, 공개 시점에 **같은 자리에서 카드 내용만 교체**된다. 사용자의 공간 기억을 훼손하지 않는다.
2. **정보 제로 상태 회피**: §9-3의 원칙("캐시된 직전 데이터가 있으면 빈 화면 대신 유지 표시")을 확장 적용한다. 완전히 빈 예고 화면 대신, **빌드 시점에 계산 가능한 실제 기술 지표 프리뷰**(KOSPI 이동평균 이격, RSI 등)를 하단에 제공해 페이지가 오늘부터 실사용 가치를 갖게 한다.
3. **절제된 예고**: 마케팅 배너식 그라데이션·일러스트는 P1(숫자가 주인공)·P2(색은 의미)에 의해 기각한다. 프리미엄함은 색이 아니라 **타이포 위계, 고스트 차트(무채색 실루엣), 정확한 로드맵 수치**로 표현한다.

### 1-2. 골격 (콘텐츠 영역)

- 사이드바·헤더는 00 문서 §6·§7 스펙을 그대로 따른다. **이 페이지의 특이사항만 기술한다.**
  - 사이드바: `Technical Analysis` 항목이 활성 상태 — 배경 `brand.primary` 10% 오버레이, 좌측 인디케이터 바 3×20px `brand.primary`, 아이콘 `brand.primary`, 라벨 `text.primary` 600 (§6-2).
  - 헤더 좌측 컨텍스트: 페이지 타이틀 `Technical Analysis` (`type.h3`). 하위 경로 없음 → 브레드크럼 미표시.
  - 듀얼 시계·검색·사용자 메뉴: 전역 스펙 그대로.
- 콘텐츠 영역: 최대폭 1600px(좌우 마진 32px, 실내폭 1536px), 12컬럼 그리드, 거터 24px, 헤더 아래 상단 여백 24px (§5-1).

### 1-3. 3단 구성 (위 → 아래)

| 행 | 구성 | 스팬 | 높이 | 역할 |
|---|---|---|---|---|
| Row 1 | **TA-HERO** (Hero 카드) | 12col | 320px | 페이지 결론 요약: "무엇이, 언제, 어떤 순서로 오는가" + 유일한 Primary CTA |
| Row 2 | **모듈 슬롯 ×3** (Standard 카드) | 4col ×3 | 각 240px | 미래 기능의 자리 예약: SLOT-01 지표 대시보드 / SLOT-02 차트 패턴 인식 / SLOT-03 백테스트 |
| Row 3 | **TA-PREVIEW** (Standard) + **TA-ROADMAP** (Standard) | 8col + 4col | 280px | 오늘 쓸 수 있는 실데이터 프리뷰 + 출시 로드맵·변경 로그 |

- 스크롤 0 상태(뷰포트 1080px − 헤더 64px = 916px)에서 Row 1 전체(320px) + Row 2 전체(240px)가 완전 노출되고 Row 3 상단(약 240px)이 걸쳐 보인다 → "아래에 실데이터가 더 있다"는 스크롤 어포던스가 자연 발생한다. P5(1화면 요약) 충족.
- 섹션 간 수직 간격은 그리드 거터와 동일한 24px(카드 간 간격 규칙)로 통일한다. Row 2와 Row 3 사이에 별도 섹션 타이틀을 두지 않는다 — 카드 헤더가 타이틀 역할을 겸한다(위계 단순화).

### 1-4. 모듈 슬롯 구조 (확장 계약)

이 페이지의 핵심 설계 자산. 향후 어떤 기능이 추가되든 아래 슬롯 규격에만 맞추면 페이지 재설계 없이 수용된다.

**슬롯 규격 3종 — 00 §8-2 카드 등급 상속 관계 명시**

`slot.L/M/S`는 00 §8-2가 정의한 Hero/Standard/Compact/Row 4등급과 별개의 새 등급이 아니다. **셋 다 예외 없이 Standard 등급(패딩 20px, 대표 타이포 `num-lg` 24px, §8-1 공통 카드 아나토미)을 상속하는 스팬 변형**이며, 서로의 차이는 오직 그리드 스팬(8/6/4col)뿐이다.

| 슬롯 등급 | 그리드 스팬 | 최소 높이 | 00 상속 등급 | 내부 패딩 | 수용 예시 |
|---|---|---|---|---|---|
| `slot.L` | 8col (약 1,016px) | 280px | Standard(와이드 변형 — 00 표준 스팬 상한 6col을 넘는 확장) | 20px | 지표 대시보드 메인 차트, 백테스트 결과 요약 |
| `slot.M` | 6col (약 792px) | 240px | Standard(표준 스팬 범위 내) | 20px | 패턴 인식 결과 리스트, 지표 히트맵 |
| `slot.S` | 4col (약 520px) | 240px | Standard(표준 스팬 하한과 동일) | 20px | 단일 지표 카드, 예고 카드(현재 상태) |

- `slot.S`가 4col이라는 이유만으로 00의 Compact(3~4col, 최소 120px, 패딩 16px)로 오인되지 않도록 명시적으로 배제한다: Compact는 "숫자 1개 + 스파크라인"만 담는 밀집 포맷이라 상태 배지("설계 중"/"개발 중")와 2~3줄 설명 텍스트를 동시에 수용할 여유가 없다. `slot.S`는 스팬만 Compact와 겹칠 뿐, 내부 콘텐츠 구성과 패딩·타이포는 처음부터 끝까지 Standard 규격을 따른다 — 이 페이지에서 Compact 등급은 아예 사용하지 않는다.
- `slot` 접두어는 **미래 기능이 자기 자리를 예약하는 모듈 카드**(SLOT-01/02/03, 그리고 데이터 역할상 TA-PREVIEW)에만 붙인다. TA-HERO(Hero 등급 그대로)와 TA-ROADMAP(스팬 확장이 없는 일반 Standard)은 슬롯 개념의 대상이 아니므로 `slot.*` 표기를 쓰지 않는다 — §3-1 총괄표의 등급 표기와 일치.
- 향후 슬롯 등급 체계를 이 페이지 밖에서도 재사용할 필요가 생기면(예: 다른 Coming Soon류 페이지), 00 문서 §8-2에 "슬롯 등급(스팬 변형)" 절을 신설해 전역 계약으로 승격하는 것을 권장한다 — 현재는 이 페이지 범위의 지역 계약으로 유지한다.

**슬롯 생애주기(상태 머신)** — 각 슬롯 카드는 아래 4단계 중 하나의 상태를 가지며, 상태별 시각 스펙이 고정된다.

| 상태 | 의미 | 시각 표현 |
|---|---|---|
| `PLANNED` | 설계 중 | 카드 배경 `surface.card`, 보더 `border.default` 1px **점선**. 내부에 고스트 콘텐츠(무채색 실루엣) + "설계 중" 회색 아웃라인 배지(`border.default`, `text.secondary` 11px). 호버·클릭 없음(비인터랙티브 → `surface.card-hover` 금지) |
| `IN-DEV` | 개발 중 | 보더 실선 복귀. "개발 중" 회색 아웃라인 배지 + 예정 분기 텍스트(`type.caption`, `text.tertiary`). 카드 클릭 시 로드맵 카드로 앵커 스크롤(1개 인터랙션만) |
| `PREVIEW` | 부분 공개 | 실데이터 일부 렌더링, §9-2 신선도 규칙 전면 적용. "프리뷰" 배지는 회색 아웃라인 유지(색 아님 — P2) |
| `LIVE` | 정식 공개 | 일반 데이터 카드로 전환. 배지 제거, 카드 전체 클릭 → 모듈 상세 뷰 |

- 현재 시점 상태: SLOT-01 `IN-DEV`, SLOT-02 `PLANNED`, SLOT-03 `PLANNED`, TA-PREVIEW는 별도 카드로서 사실상 `PREVIEW` 상태의 선행 구현.
- 슬롯이 `LIVE`로 전환되면 해당 카드는 Row 2의 4col 예고 카드에서 Row 3 이하의 `slot.L`/`slot.M` 본 배치로 승격되고, Row 2의 원 위치에는 다음 예정 모듈이 채워지거나 Row 2 자체가 축소된다(§7. 개선 포인트에서 전환 절차 상술).

### 2-2. TA-PREVIEW 카드 — 세션 상태 2종 (§9-4 세션 인식 렌더링)

프리뷰 데이터는 1일 1회 빌드 배치 산출물이므로 **장중에도 라이브가 아니다.** 라이브로 오독되지 않도록 두 상태를 명시적으로 분리한다.

```
[상태 A — 장중(09:00–15:30 KST) 열람 시]                [상태 B — 데이터 STALE(빌드 실패·경과 ≥ 3T) 시]
┌─ 기술 지표 프리뷰 — KOSPI ──────────────────┐  ┌─ 기술 지표 프리뷰 — KOSPI ──────────────────────┐
│ 07/09 마감 기준          [지표 산출 안내 ⓘ]  │  │ 07/07 마감 기준    [오래된 데이터] (yellow 12%)  │
│ ────────────────────────────────────────── │  │ ─────────────────────────────────────────────── │
│  2,674.31   ▲ +1.25%   58.4   상승 유지     │  │  2,668.90   ▲ +0.84%   55.1   상승 유지          │
│  (text.primary 유지 — CLOSED-SNAPSHOT)      │  │  (값 전체 text.secondary로 강등 — STALE)         │
│  라이브 점·펄스 없음. "마감 기준" 라벨 필수    │  │  툴팁: "마지막 갱신 07/08 07:12 KST"             │

└─────────────────────────────────────────────┘  └──────────────────────────────────────────────────┘
```

- 상태 A: 값은 `text.primary` 유지 + "07/09 마감 기준" 라벨 필수, `badge.live`·펄스 전면 금지 (S2).

- 등락률에는 항상 기준가를 병기한다 — 카드 푸터 캡션 "산출 기준: 전일 정규장 종가" (S3).

### 4-2. 시나리오 B — 저녁 종목 복기 중 기술 지표 확인 (체류 1~2분)

### 4-3. 시나리오 C — 재방문자의 로드맵 확인 (체류 15초)

## 5. 디자인 의도

1. **"빈 페이지"를 "예약된 페이지"로 재정의**: Coming Soon을 별도의 특수 화면으로 만들지 않고, 미래 화면의 그리드에 상태(PLANNED/IN-DEV)만 입힌 형태로 설계했다. 기능 공개는 "화면 교체"가 아니라 "카드 상태 전환"이 된다. 사용자에게는 연속성, 팀에게는 재설계 비용 제로.
2. **날짜를 주인공 숫자로**: P1(숫자가 주인공)은 시세가 없는 페이지에서도 유효하다. 이 페이지에서 사용자가 가장 알고 싶은 수치는 "언제 쓸 수 있나"이므로 `2026 Q3`를 `type.num-display` 32px로 격상했다. 장식 헤드라인 대신 수치가 결론을 전달한다.
3. **신뢰는 진짜 데이터에서 나온다**: 예고만 있는 페이지는 재방문 동기가 없다. TA-PREVIEW는 이미 매일 생성되는 종가 데이터로 계산 가능한 지표 4종을 제공해, 페이지가 오늘부터 루틴에 편입될 최소 근거를 만든다. 동시에 §9 신선도 규칙을 완전 적용해 "프리뷰조차 as-of가 정확한 제품"이라는 브랜드 신뢰를 축적한다.
4. **고스트 콘텐츠의 절제**: 미래 기능 미리보기를 유채색 목업으로 그리면 실데이터로 오독된다(야간선물 사고와 동일 구조의 위험). 모든 미리보기는 `skeleton.*` 2톤 무채색 + "예시" 캡션으로 통일해, **색이 있는 숫자 = 진짜 데이터**라는 전역 계약을 이 페이지에서도 깨지 않았다. Hero의 미리보기 패널(§3-2)은 그 역방향 사례다 — 이번엔 진짜 데이터인데도 일부러 무채색으로 눌러, "이 화면의 색 예산은 TA-PREVIEW 카드 하나에만 배정돼 있다"는 신호를 지켰다. 같은 원칙을 반대 방향에서 적용해도 결과가 일관되게 나온다는 점이 이 계약의 견고함을 보여준다.
5. **기대 관리의 정직함**: 로드맵에 분기 단위 일정과 변경 로그를 상시 노출한다. 일정이 밀리면 로그에 남는다. Bloomberg/Linear류 도구 사용자가 기대하는 태도 — 과장 없는 상태 보고 — 를 예고 페이지에서부터 구현한다.

## 6. 색상 및 여백 사용 이유

### 6-1. 색상

- **이 페이지의 의미 색은 최대 3종**으로, R1 상한(4종) 아래로 유지된다: ① `market.up` — TA-PREVIEW의 이격률 1곳 ② `brand.primary` — Hero CTA 1개 + 로드맵 진행 점 + 사이드바 활성 ③ `accent.yellow` — STALE/지연 배지(발생 시에만). 화면 기본 상태에서는 사실상 유채색 2종만 존재한다.
- **Hero 미리보기 패널(§3-2)은 실데이터를 담지만 새 색을 추가하지 않는다**: 종가·RSI 값을 `text.secondary` 무채색으로만 표기해 3색 예산에 4번째 색을 얹지 않았다. 색 예산은 항상 TA-PREVIEW 카드 본체(§3-5)가 전담하고, Hero의 요약 인용은 끝까지 조용하게 둔다 — 같은 숫자라도 위치(Hero 요약 vs 본 카드)에 따라 색 처리를 다르게 가져가는 것이 아니라, "실데이터 표시 = 색"이라는 원칙 자체를 지키되 그 색을 **1곳에서만** 켜서 중복 강조를 막는다.
- **개발 상태 배지를 회색 아웃라인으로 한 이유**: "개발 중"에 노랑을 쓰면 `status.warning`(지연 데이터)과 의미가 충돌한다. 상태 배지의 노랑은 데이터 신선도 전용으로 보존하고, 제품 개발 단계는 색이 아닌 텍스트+보더로 구분한다(P2 규칙 문장 그대로 적용).
- **고스트 프리뷰의 완전 무채색**: 미래 기능 실루엣에 R5 차트 팔레트를 쓰지 않았다. 유채색 시리즈는 "지금 유효한 데이터"라는 학습을 사용자에게 시키는 중이므로, 예시 그래픽이 그 학습을 오염시키면 안 된다.
- **PLANNED 슬롯의 점선 보더**: 톤을 낮추는 대신 보더 스타일 하나로 상태를 구분했다(P4 — 구분은 보더로). 배경을 어둡게 하거나 투명도를 낮추면 대비 미달로 접근성(§11-1)이 깨지기 때문이다.
- **TA-PREVIEW 차트 시리즈에서 #F87171 제외**: 이 카드에는 등락 색 텍스트(▲ +1.25%)가 공존하므로, 라인 색으로 빨강 계열을 쓰면 "하락 신호"로 오독될 수 있다. R5 팔레트 중 #60A5FA + #6B7280만 선택하고 라벨을 직접 표기했다.

### 6-2. 여백

- **4pt 스케일 엄수**: 카드 간·행 간 24px, Hero 패딩 24px, Standard 패딩 20px, 라벨↔값 4px — 전부 §4-1 표 값이며 임의 값이 없다.
- **Hero 내부의 넉넉한 48px 스탯 간격**: 정보량이 적은 예고 화면에서 요소를 흩뿌리면 빈약해 보인다. 대신 스탯 3개를 48px 간격으로 묶어 좌측 하단에 밀도를 만들고, 우측은 미리보기 패널(180px) + 고스트 차트(420px) 두 블록으로 나눠 채워 "성긴 화면"이 아니라 "여백이 설계된 화면"으로 읽히게 했다. 두 블록으로 쪼개기 전(고스트 차트 620×248px 단독)에는 무채색 실루엣 하나가 카드 절반 가까이를 차지해 정보 밀도 대비 여백이 과했으므로, 폭 200px가량을 실데이터 미리보기로 되돌려 배분했다.
- **Row 2의 균등 3분할(4col ×3)**: 세 모듈의 서열을 시각 크기로 매기지 않았다. 서열은 로드맵의 시간축(분기)이 담당하고, 그리드는 평등하게 유지해 향후 어떤 모듈이 먼저 LIVE가 되어도 배치 논리가 깨지지 않는다.
- **스크롤 0 컷라인 활용**: Row 3이 약 240px만 걸쳐 보이도록 높이를 배분(320+24+240+24)했다. 프리뷰 카드의 헤더와 첫 수치가 컷라인 위에 노출되어, 별도 화살표나 애니메이션 없이(모션 규칙 §10-2 준수) 스크롤을 유도한다.

## 7. 개선 포인트

1. **슬롯 LIVE 전환 절차의 표준화**: 모듈 공개 시 ① Row 2 슬롯 카드가 LIVE 스타일로 전환(공개 후 7일간 헤더에 "신규" 회색 아웃라인 배지) ② 다음 빌드에서 해당 모듈이 Row 3 이하의 `slot.L`/`slot.M` 본 배치로 승격 ③ Hero 핵심 수치가 "공개 모듈 n / 3"으로 갱신 — 이 3단계를 페이지 계약으로 문서화해 어떤 모듈이든 동일하게 적용한다.
2. **프리뷰 지표의 점진 확장**: 정적 빌드 환경에서도 전일 종가 시계열만으로 계산 가능한 지표가 많다(스토캐스틱, OBV, 52주 위치, 이격도 밴드). 분기마다 1~2종씩 TA-PREVIEW에 추가하면 Coming Soon 기간 자체가 기능 성장기가 된다. Hero의 "프리뷰 지표 n종" 수치와 자동 동기화.
3. **종목 단위 프리뷰**: 현재는 KOSPI 단일 대상이다. 관심종목 상위 3종(예: 삼성전자, SK하이닉스, NVIDIA)의 20일선 이격·RSI를 Row 규격(행 52px)으로 TA-PREVIEW 하단에 추가하는 확장을 SLOT-01 공개 전 중간 단계로 검토한다.
4. **관심 기능 투표**: 서버가 없으므로 투표 집계는 불가하다. 대안으로 각 PLANNED 슬롯에 "이 기능이 먼저 필요해요" 토글(localStorage 저장)을 두고, 선택 시 해당 슬롯 우선 안내를 로컬에서만 개인화하는 수준으로 구현한다. 집계 기반 우선순위 결정은 향후 이슈 트래커(GitHub Discussions) 링크로 대체.
5. **정적 환경 제약의 명시**: TA-PREVIEW의 신선도 판정은 빌드 시 기록된 as-of 타임스탬프와 열람 시점 클라이언트 시계의 비교로 수행된다(§12 전제와 동일). 데이터 파일에는 값·생성 시각·산출 기준(전일 종가)이 항상 포함되어야 하며, 빌드 실패 시 직전 파일을 유지해 STALE 상태로 노출한다 — 빈 카드 금지.
6. **로드맵 알림 연계(§3-6 정적 텍스트의 전제 조건)**: 현재 TA-ROADMAP 변경 로그 행은 목적지가 없어 정적 텍스트로 고정돼 있다. 03(News) 페이지의 4개 탭(미국시장/한국시장/거시경제/속보)에 제품 업데이트·공지 전용 카테고리(탭 내 필터 또는 별도 배지)를 신설하면, 모듈 공개·일정 변경을 그 카테고리로 발행하고 TA-ROADMAP 변경 로그가 해당 공지로 링크하도록 §3-6 규칙을 갱신한다(클릭·호버 배경 활성화, Tab 포커스 순서에 재편입). 별도 알림 인프라 없이 기존 뉴스 파이프라인을 재사용하는 정적 환경 친화적 설계이며, News 쪽 카테고리 신설은 03 문서의 변경 범위다.
7. **백테스트 모듈의 사전 설계 부채**: SLOT-03(백테스트)은 계산량이 커 클라이언트 실행 또는 빌드 시 사전 계산 중 택일이 필요하다. 슬롯 규격(`slot.L`, 280px+)은 결과 요약 카드 기준으로 예약되어 있으므로, 상세 실행 UI는 페이지 내 확장이 아닌 별도 하위 뷰(3클릭 원칙 내 진입)로 설계하는 것을 전제한다.
