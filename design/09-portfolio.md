# 09. Portfolio

**AI Trading Assistant — Portfolio 페이지 설계 v1.0**
기준 문서: 00. 디자인 시스템(Global Foundation v1.0). 본 문서는 00의 토큰명·수치를 변경 없이 인용한다.
페이지 성격: 🔒 비밀번호 게이트 뒤에 있는 **계좌 단위 포지션 관리 화면**. Asset 페이지(자산 전체의 시계열 추이·배분)와 역할을 분리하여, Portfolio는 "지금 각 계좌에 무엇을 얼마나 들고 있고, 얼마를 벌고 있는가"에만 집중한다.

## 1. 전체 레이아웃 설명

### 1-1. 페이지 골격

00 문서 §5-1의 공통 골격(사이드바 240px + 헤더 64px + 콘텐츠 최대폭 1600px, 12컬럼/거터 24px/좌우 마진 32px)을 그대로 사용한다. 본 페이지에서의 특이사항만 기술한다.

- **사이드바**: `Portfolio` 항목 활성(좌측 3×20px `brand.primary` 인디케이터 바). 게이트 해제 후에는 §6-3 규칙에 따라 자물쇠 아이콘이 **열린 자물쇠 14px `text.tertiary`** 로 교체된다. Asset과 Portfolio는 **동일 게이트 세션을 공유**하므로, 한쪽에서 해제하면 두 항목의 자물쇠가 동시에 열린 상태로 표시된다.
- **헤더**: 페이지 컨텍스트는 `Portfolio / 키움`처럼 **현재 선택된 계좌 탭을 브레드크럼으로 병기**한다(`type.h3` + `text.tertiary` 12px). 탭 전환 시 브레드크럼도 즉시 갱신 — 헤더만 봐도 어느 계좌를 보는 중인지 알 수 있다.
- **콘텐츠 수직 구조 (위 → 아래)**:
  1. **Hero — 포트폴리오 종합 카드** (12col): 전 계좌 원화 환산 총 평가액·일간/누적 손익·계좌별 요약·통합 자산 스택 바. "3초 안에 핵심 파악"(P5)을 담당하는 층.
  2. **계좌 탭 바** (12col, 높이 44px): 키움 / 한투 ISA / 한투 위탁 / BYBIT — 4개 고정 탭. 각 계좌의 운용 역할은 단타·스윙 / ETF / 미국주식 / 암호화폐로 고정되어 있으며, 탭 툴팁과 계좌 요약 카드 라벨에 이 역할을 병기한다.
  3. **계좌 요약 Compact 카드 4장** (3col ×4): 계좌 평가액 / 예수금 / 평가손익 / 전체 자산 대비 비중.
  4. **보유종목 테이블** (8col) + **계좌 내 비중 구성 카드** (4col).
  5. **종목 상세 패널(Position Detail Panel)** (우측 슬라이드 드로어 480px, Level 2): 보유종목 행 클릭 시 등장. Stock Hub 전역 컴포넌트의 5개 블록을 통째로 포함하는 확장판이며(§1-3), 스크롤 흐름 밖의 오버레이 층이다.

### 1-2. 잠금 게이트 동작

- 미해제 상태로 진입하면 콘텐츠 영역에 `overlay.scrim`(#000000 60%)과 **비밀번호 게이트 모달(Level 3, 폭 400px)** 을 띄운다. 스크림 뒤에 실제 금액을 절대 렌더링하지 않는다(블러 처리로 실값을 가리는 방식 금지 — 소스 노출 위험).
- **마스킹 스켈레톤을 두지 않는 이유(★리뷰 반영 — Asset 초기 설계와의 차이)**: 60% 불투명 검정 스크림 위에서는 배경에 깔아둔 스켈레톤 블록의 형태가 거의 지각되지 않는다. 지각되지 않는 골격을 사전에 그려 두는 것은 "해제 후 점프 없음"이라는 목적에 비해 렌더링 복잡도만 키우는 비효율이다. 따라서 Portfolio는 스크림 아래를 완전한 빈 `surface.canvas` 상태로 두고, 대신 **해제 애니메이션 자체가 점프 없음을 보장**하도록 설계를 바꾼다: 모달 퇴장(페이드 아웃 120ms) → 콘텐츠 전체(Hero·탭 바·카드)가 이미 최종 위치에 배치된 채 8px 아래에서 위로 이동하며 페이드 인(180ms, `motion.base`)한다. 요소가 위치를 바꾸는 것이 아니라 불투명도만 바뀌므로 레이아웃 시프트가 발생하지 않는다.
- 비밀번호 검증·재잠금·세션 공유 등 **게이트의 동작 로직 자체는 Asset 페이지와 완전히 동일**하며, 차이는 오직 잠금 화면의 배경 렌더링 방식(스켈레톤 유무)에 한정된다. Asset도 동일한 근거(스크림 위에서 골격이 지각되지 않음)가 성립하므로, 다음 Asset 개정 시 동일하게 스켈레톤을 제거해 두 게이트의 시각 언어를 다시 맞출 것을 제안한다(§7).
- 모달 구성: 자물쇠 아이콘 32px `text.tertiary` → 타이틀 "Portfolio 잠금 해제" `type.h2` → 설명 1줄 `type.body` `text.secondary` → 입력 필드 1개(높이 40px, `surface.sunken`, `border.default`, radius 8px) → **확인 버튼(Primary Green 채움, 이 화면 유일 CTA — R3 충족)**.
- 실패 시: 필드 보더 `status.danger` + 아래 헬퍼 텍스트 12px `status.danger` "비밀번호가 일치하지 않습니다". 흔들림 애니메이션 없음(§6-3).
- 해제 성공 시: 모달 퇴장 → 위의 페이드+상승 애니메이션으로 콘텐츠 등장. 레이아웃 점프 없음(§9-1)은 스켈레톤이 아니라 애니메이션 계약으로 보장한다. 세션 종료 시 재잠금.

### 1-3. Stock Hub 전역 컴포넌트와의 관계 — ★명시적 정의(★리뷰 반영 — 재사용 범위·타이포·클릭 수 정합)

Portfolio는 종목 관련 UI를 **재발명하지 않는다.** 관계는 다음 한 문장으로 고정한다:

> **종목 상세 패널 = Stock Hub 전역 컴포넌트의 5개 블록(헤더·시세개요·액션 허브·다가오는 일정·관련 뉴스, 05 §3-1 총괄표 그대로)을 '변경 없이' 재사용 + Portfolio 전용 'PositionBlock' 1종을 그 사이에 끼워 넣은 확장(Composition).**

| 컴포넌트 | 출처 | Portfolio에서의 사용 방식 |
|---|---|---|
| `SecurityHeader` (패널 헤더 + 시세 개요: 종목명·즐겨찾기·티커/시장·현재가 `num-display` 32/700·등락·세션 배지·as-of·스파크라인·3×2 스탯 그리드[거래대금·거래량·시가총액·52주 최고·52주 최저·PER]) | Stock Hub 전역(05 §3-1 A·B 블록) | **변경 없이 재사용.** 세션 인식 렌더링(§9-4)·스크롤 시 컴팩트 헤더 축약(05 §2-4) 로직까지 그대로 상속 |
| `ActionHub` ([▶ TradingView 차트 ↗] / [재무제표 분석 →] 버튼 2개) | Stock Hub 전역(05 §3-1 C 블록) | **변경 없이 재사용.** 패널이 열린 동안 화면에서 유일한 Primary 채움 버튼(R3) |
| `NewsRow` (헤드라인·출처·시각 행) | Stock Hub 전역(05 §3-1 E 블록) | **변경 없이 재사용.** 해당 종목 필터만 적용, 자체 푸터("관련 뉴스 전체 보기 →") 포함 |
| `EventRow` (날짜·유형 배지·제목·D-day 행) | Stock Hub 전역(05 §3-1 D 블록) | **변경 없이 재사용.** 단, 보유자 맥락에 맞춰 유형 필터를 6종으로 재정의(§3-6 참조), 자체 푸터("전체 일정 보기 →") 포함 |
| `PositionBlock` (매입가·수량·수익률·실제 가치) | **Portfolio 전용 신규** | 계좌 보유 정보 레이어. Stock Hub에는 존재하지 않으며, ActionHub와 관련 일정 사이에 한 겹만 끼워 넣는다(§3-6) |

- 보유종목 테이블의 행 역시 §3-4의 **Row 등급(52px)** 을 그대로 쓰되, 열 구성만 보유 정보(매입단가·수익률·비중)로 확장한 변형이다.
- **05 §1-1 트리거표와의 정합**: 05는 "Portfolio/Asset: 보유종목 Row 클릭"을 Stock Hub의 호출 지점 중 하나로 기술한다. 위 표가 확정하듯 Portfolio의 패널은 Stock Hub 5개 블록 전부를 원본 그대로 포함하는 **상위집합(superset)**이지, 별개의 축약 컴포넌트가 아니다. TradingView·재무제표 버튼이 ActionHub로 그대로 패널 안에 들어와 있으므로, 종목 상세를 연 시점부터 심층 분석까지 **추가 클릭 0**으로 도달한다 — Dashboard·News·Stock에서 Stock Hub를 여는 것과 동일한 클릭 비용 구조이며, 05 §4-5의 3클릭 검증표와도 어긋나지 않는다.
- 이전 초안에서 패널 푸터에 두었던 "Stock Hub에서 전체 분석 →" 단일 링크는 제거한다. ActionHub가 이미 TradingView·재무제표로의 직행 경로를 제공하고, 뉴스·일정 블록도 05 원본의 자체 푸터를 그대로 상속하므로 별도의 통합 링크가 오히려 중복이자 혼동 요소였다.

### 1-4. 통화·세션 처리 원칙

- **합계·비중 계산은 전부 원화 환산**으로 통일하고, Hero에 "환율 1,378.50원 적용" 캡션을 병기한다.
- **개별 종목 행은 상장 시장 통화**를 따른다(§3-3): 키움/한투 ISA 행은 `₩84,300`, 한투 위탁(미국주식) 행은 `$212.40`, BYBIT 행은 `$67,240.50`. 한 행 안에서 단위를 섞지 않는다.
- 세션 인식(§9-4): 국내 계좌 탭은 KRX 세션 기준, BYBIT 탭은 24/7 세션. 국내 장 마감 후 국내 탭의 모든 시세·평가액은 **CLOSED-SNAPSHOT**("07/10 15:30 마감 기준") 스타일로 전환되고 라이브 점·펄스가 전면 금지된다. BYBIT은 상시 세션이므로 데이터가 FRESH일 때만 라이브 점을 허용한다.

- 금액 가리기 토글 ON 시 모든 절대 금액을 `₩•••,•••`로 마스킹하고 %·등락률은 유지. 상태는 게이트 세션과 함께 저장.
- Hero의 계좌 행 클릭 = 해당 계좌 탭으로 즉시 전환(스크롤 이동 포함). 통합 스택 바 세그먼트 호버 시 툴팁(Level 2)으로 계좌명·금액·비중.
- **계좌별 자산 배분 색 처리 — Asset과 통일(★리뷰 반영)**: 이 스택 바는 Asset 페이지 §3-5 "자산 배분" 카드와 완전히 동일한 정보(계좌 비중)를 다루는 동일 위젯이다. 같은 게이트 뒤에서 같은 정보를 담은 위젯이 페이지마다 우연한 색 예산 상황(어느 페이지가 먼저 4색을 다 썼는지)에 따라 전혀 다른 모습으로 그려지면, 제품 전체의 시각 언어가 일관성을 잃는다. 이에 Portfolio Hero도 Asset과 동일하게 **회색 3단계 교차(#4B5563 / #374151 / #6B7280) + 세그먼트별 라벨 직접 표기**로 고정한다. 이 규칙은 "계좌"처럼 개수가 4개로 고정된 카테고리 배분에 적용되며, 계좌 *내부* 종목처럼 개수가 가변적인 다중 시리즈 배분(§3-5)에는 적용되지 않는다 — 두 위젯의 구분 기준은 §3-5와 §6-1에서 상세히 다룬다.
- 장기적으로는 이 스택 바를 00 문서에 전역 컴포넌트(예: `AccountAllocationBar`)로 승격해 Asset·Portfolio가 각자 스펙을 복제하지 않고 하나의 정의를 상속하도록 정리할 것을 제안한다(§7).

- **파이/도넛 차트를 채택하지 않은 이유**: 5~10개 세그먼트의 각도 비교는 길이 비교보다 정확도가 낮고, 카드 폭 496px에서 도넛+범례는 라벨 직접 표기(11-2 색맹 대응 원칙)를 어렵게 한다. 가로 스택 바 + 순위 리스트는 같은 데이터를 "비교(바) + 정밀값(리스트)" 2단으로 제공한다.
- 스택 바는 등락 정보가 아니므로 ▲▼·부호를 절대 붙이지 않는다 — `market.down`과 동일한 #60A5FA가 팔레트에 존재하기 때문에, "부호·기호 없음 = 등락 아님"이라는 R4의 역방향 신호로 오독을 차단한다.
- **Hero의 '통합 자산 구성'과의 구분(★리뷰 반영 — 위젯 스코프 명확화)**: 이 카드는 계좌 *간* 비중이 아니라 한 계좌 *안*의 종목별 비중을 다룬다 — Hero 스택 바(§3-1, 계좌 5개로 개수 고정, Asset과 동일한 무채색 규칙 상속)와는 서로 다른 위젯이다. 여기서는 보유 종목 수가 가변적이고(계좌마다 7~15개 이상) 개별 종목을 시각적으로 구분해야 하므로, R5가 정의한 "차트 시리즈 색"의 용도에 정확히 해당한다 — 5색 고정 팔레트 + 6번째 이후 회색 묶음 규칙을 그대로 쓴다. 이 카드가 Hero와 다른 색 처리를 쓰는 것은 우연한 불일치가 아니라, 두 위젯이 다루는 데이터 성격(개수 고정 카테고리 vs 가변 다중 시리즈)이 근본적으로 다르기 때문이다.

## 5. 디자인 의도

1. **"총액 → 계좌 → 종목 → 근거(뉴스·일정)"의 4단 깔때기.** Hero(전체) → 탭+Compact(계좌) → 테이블(종목) → 패널(근거)로 정보 밀도가 단계적으로 상승한다. 각 단계는 이전 단계를 가리지 않는다 — 상세 패널조차 드로어 형식이라 테이블 문맥이 화면에 남는다. Bloomberg식 밀도를 유지하되 "한 번에 한 층만 깊어지는" Apple식 위계로 복잡도를 제어했다.
2. **Asset과의 역할 분리.** Asset = 시간축(자산이 어떻게 변해왔나), Portfolio = 단면(지금 무엇을 들고 있나). 같은 게이트 뒤에 있지만 Hero에 추이 차트를 넣지 않은 것은 의도적 중복 회피다.
3. **Stock Hub 컴포넌트의 '재사용 + 확장' 원칙, 그리고 그 재사용은 문자 그대로여야 한다.** 종목 시세·뉴스·일정·차트/재무제표 진입 UI를 Portfolio가 따로 만들면 세션 인식(§9-4) 같은 안전 규칙이 두 벌로 갈라져 사고 재발 위험이 생긴다. SecurityHeader뿐 아니라 ActionHub까지 그대로 상속하는 순간 야간선물류 사고 방지 계약과 3클릭 도달 계약(05 §4-5)이 동시에 자동 상속된다. 이전 설계에서 ActionHub를 빼고 별도 링크로 대체했던 것은 재사용 범위를 스스로 좁혀 이 이점을 반쯤 포기하는 선택이었다 — 이번 개정에서 바로잡는다. Portfolio 고유 가치는 PositionBlock 한 겹뿐이며, 그래서 유지비가 가장 낮은 구조다.
4. **violet 해석 레이어를 이 페이지에서는 의도적으로 쓰지 않는다.** 집중도 경고("상위 1종목 쏠림 주의"), 계좌 등락 요약, 환율 적용 안내 등은 모두 `accent.violet` 인사이트 박스를 붙이고 싶어지는 후보 지점이다. 그러나 Dashboard·Macro·News·Financial Statements·Asset을 합쳐 이미 5개가 넘는 페이지에서 동일한 "좌측 violet 바 + 회색 캡션" 형태가 반복적으로 등장하고 있어(제품 전반의 색 사용 리뷰), violet 박스 자체가 정형화된 위젯으로 소비되며 "이건 시스템이 짚어준 특별한 판단"이라는 본래의 신호력을 잃어가고 있다. Portfolio는 이 신호력을 되살리는 쪽에 서기로 한다 — 위 후보 지점은 모두 `status.warning` 텍스트 또는 일반 캡션(`text.tertiary`)으로만 표현하고(§3-5, §6-1), violet은 정말 시스템이 해석을 개입시키는 핵심 지점(Dashboard 요약, Financial Statements 진단)에만 남긴다.
5. **게이트는 '문'이지 '벽'이 아니다.** 다만 그 문이 열리는 방식은 배경 골격이 아니라 애니메이션 계약으로 보장한다(§1-2) — 60% 스크림 위의 스켈레톤은 지각되지 않아 실효가 없으므로, 페이드+상승 전환만으로 점프 없음을 달성해 불필요한 렌더링 복잡도를 없앴다. 실데이터는 스크림 뒤에도 렌더링하지 않아 프라이버시를 지킨다. 금액 가리기 토글은 카페 등 공공장소 열람이라는 실사용 맥락에 대응한다.
6. **비중 시각화는 '비교'와 '정밀값'을 분리 제공.** 스택 바(길이 비교) + 순위 리스트(정확한 %) + 테이블 마이크로 바(행 단위 문맥)의 3중 표현은 같은 데이터를 용도별로 나눈 것이지 장식 중복이 아니다. 집중도 경고 캡션은 "화면이 결론을 먼저 말한다"(P5)의 적용이다. 계좌 간 배분(Hero)과 계좌 내 종목 배분(§3-5)을 서로 다른 색 문법으로 명확히 분리한 것도 같은 맥락 — "비교 대상의 성격이 다르면 시각 언어도 의도적으로 달라야 한다"는 원칙이며, 우연한 불일치와는 다르다.
7. **일정 6종 고정은 신호 대 잡음비 설계다.** 실적·공시·배당·IR·규제·락업은 보유자의 의사결정(보유 지속/축소)에 직접 연결되는 이벤트만 추린 화이트리스트다. 필터 UI로 열어두지 않고 데이터 계약으로 잠근 것은, 잡음 이벤트가 D-day 배지의 긴급성을 희석하는 것을 막기 위함이다.

## 6. 색상 및 여백 사용 이유

### 6-1. 색상

- **이 페이지의 의미 색 상한(R1) 검증**: 등락 2색(`market.up` #F87171 / `market.down` #60A5FA) + 상태 1색(`status.warning` — 지연 배지·쏠림 경고·D-7 강조) + CTA 1색(`brand.primary` — 게이트 확인 버튼, 그리고 패널이 열려 있는 동안은 ActionHub의 TradingView 버튼·탭 인디케이터가 공유하는 동일 토큰) = 4종. 게이트 버튼과 ActionHub 버튼은 화면 상태(잠김 / 해제 후 패널 열림)가 서로 겹치지 않으므로 두 개의 Primary 채움이 동시에 존재하는 일은 없다(R3 유지). §3-5 계좌 내 비중 카드의 차트 팔레트는 R1이 아니라 R5(차트 시리즈 색, 별도 예산)에 속하며 등락·상태와 화면 영역이 분리된다.
- **accent.violet 미사용(★리뷰 반영)**: §5-4에서 설명한 대로, 이 페이지는 해석 레이어 박스를 의도적으로 쓰지 않는다. 후보 지점(집중도 경고 등)은 모두 `status.warning` 텍스트 또는 일반 캡션으로 대체했다 — violet의 신호력을 Dashboard·Financial Statements 등 핵심 지점에 남겨두기 위한 페이지 차원의 절제이며, 위 4종 예산에는 애초에 포함되지 않는다.
- **손익은 '변화값'이므로 등락색을 쓴다**: 평가손익·수익률은 §3-3의 "변화값(등락)만 `market.*` 색" 조항에 정확히 해당한다. 반면 평가액·예수금·매입단가 같은 절대값은 전부 `text.primary` — 화면에서 색이 있는 숫자는 "방향이 있는 숫자"뿐이라는 단일 문법이 성립한다.
- **계좌별 자산 배분(Hero)을 무채색으로 통일한 이유(★리뷰 반영)**: 이 스택 바는 Asset 페이지 §3-5 "자산 배분" 카드와 정확히 같은 정보(계좌 비중)를 다루는 동일 위젯이다. 같은 게이트 뒤, 같은 정보를 두 페이지가 정반대 색 언어(5색 팔레트 vs 완전 무채색)로 그리면 제품의 시각 언어가 페이지별 우연한 색 예산 상황에 좌우된다. 따라서 Hero의 통합 자산 구성도 Asset과 동일하게 **회색 3단계 교차(#4B5563 / #374151 / #6B7280) + 라벨 직접 표기**로 고정한다. 이 규칙은 계좌처럼 개수가 고정된 카테고리(4개 계좌)에 적용되며, §3-5의 계좌 *내부* 종목 비중처럼 개수가 가변적인 다중 시리즈에는 여전히 R5 5색 팔레트가 적용된다 — 두 위젯은 이름이 비슷할 뿐 데이터 성격이 다르다(§3-1·§3-5 참조).
- **#60A5FA 이중 사용(하락색 ↔ §3-5 팔레트 2번)의 충돌 관리**: Hero의 계좌 배분이 무채색으로 바뀌면서 이 충돌은 §3-5 계좌 내 비중 카드 한 곳으로 범위가 좁혀졌다. 그 안에서도 스택 바·색 칩에는 ▲▼·±부호를 절대 병행하지 않는다 — R4에 의해 "부호+색 = 등락, 색 단독+라벨 = 구성 비중"으로 채널이 갈라져 오독 여지가 없다. 추가로 스택 바와 등락 수치 사이에는 최소 16px 이상의 수직 간격과 캡션 라벨을 배치했다.
- **예수금 세그먼트를 `skeleton.base` 무채 저명도로 처리한 이유**: 현금은 "아직 배치되지 않은 자산"이다. 유채색 종목 세그먼트와 명도 대비로 구분하면 색 하나를 아끼면서 의미(비어 있음)도 전달된다 — R2(회색 단계 우선)의 적용.
- **일정 유형 배지를 회색 아웃라인으로 통일한 이유**: 유형이 6종이라 색으로 구분하면 그 자체로 R1을 초과한다. `type.overline` 텍스트가 1차 정보, 형태(캡슐)가 2차 — 색맹 사용자에게도 동일한 정보량이다(§11-2).
- **잠금 상태 표현에 색을 쓰지 않은 이유**: 자물쇠·톤 다운(`text.tertiary`)만으로 충분하며, 노랑·빨강을 쓰면 "경고/오류"로 오독된다. 잠김은 이상 상태가 아니라 정상 상태다.

### 6-2. 여백

- **카드 간 24px(거터 동일)**: Compact 4장과 테이블·비중 카드가 같은 리듬으로 정렬되어, 정보량이 많아도 격자 규칙성이 "복잡해 보임"을 상쇄한다(0-1의 모순 해법).
- **섹션 간 32px**: Hero ↔ 탭 바 ↔ 카드 군 사이는 24px이 아닌 32px — "층이 바뀐다"(전체→계좌)는 신호를 여백 폭 차이로 전달한다.
- **라벨↔값 4px, 값↔등락 4px**: PositionBlock과 Compact 카드에서 라벨-값 쌍을 4px로 밀착시켜 하나의 의미 단위로 읽히게 하고, 쌍과 쌍 사이는 16px 이상 벌려 그룹 경계를 만든다(근접성 원리).
- **행 52px 유지**: 보유종목이 7~15개 수준인 개인 계좌에서 밀집 40px은 불필요하다. 52px은 마이크로 바·등락 배지가 들어가도 터치·클릭 히트 영역 32px(§11-3)을 여유 있게 확보한다.
- **패널 480px**: 뉴스 헤드라인 1줄(약 30자)과 2열 PositionBlock이 말줄임 없이 들어가는 최소폭이며, 1920px에서 테이블 좌측 열(종목명~현재가)이 가려지지 않아 문맥 유지가 된다. 스탯 그리드·ActionHub가 추가되어 패널 총높이는 늘었지만(§2-3), 폭 정책은 05 원본과 동일하게 480px로 고정해 컴포넌트 이식성을 지켰다.

## 7. 개선 포인트

1. **통합 이벤트 캘린더**: 현재는 종목 패널 단위로만 일정을 본다. 전 보유종목의 [실적][배당][락업] 6종을 월 캘린더 뷰(12col Standard 카드)로 묶은 "이번 달 내 포지션 일정" 섹션을 Hero 아래 접이식으로 추가하면 시나리오 4가 3클릭 → 1클릭이 된다.
2. **목표 비중 대비 리밸런싱 뷰**: 비중 구성 카드에 사용자가 설정한 목표 비중(예: 삼성전자 15%)을 고스트 마커로 오버레이하고, 이탈 ±5%p 초과 시 `status.warning` 캡션. 정적 환경에서도 로컬 설정만으로 구현 가능하다.
3. **실현손익 레이어**: 현재 설계는 평가손익 중심이다. 매매일지 데이터(이미 생성 중)와 조인해 계좌별 실현손익·수수료·세금 추정 탭을 추가하면 "실제 가치"의 정확도가 올라간다.
4. **환율 변동 분해**: 한투 위탁·BYBIT의 원화 환산 손익을 "가격 요인 vs 환율 요인"으로 분해한 툴팁 제공 — 해외 자산 비중이 커질수록 필요해진다.
5. **스냅샷 히스토리와 Asset 연계**: 빌드 시마다 총 평가액 스냅샷을 누적하면 Asset 페이지의 추이 차트 원천 데이터가 된다. Portfolio에는 차트를 두지 않고 데이터 파이프만 공유해 역할 분리를 유지한다.
6. **구현 제약 관련(간단히)**: 정적 GitHub Pages 특성상 ① 시세·평가액은 빌드 시점 값이므로 §9-2 신선도 배지가 품질의 핵심 방어선이다 — 보유종목 데이터 파일에도 as-of·유효 세션 메타를 필수 포함한다. ② 비밀번호 게이트는 열람 차단(프라이버시 가림) 수준이며 이 한계는 Settings 고지 문구(§12)로 일관되게 안내한다. ③ 계좌 잔고의 자동 동기화(키움 OpenAPI 등)는 로컬 빌드 단계에서의 수집으로 한정하고, 실패 시 §9-3 원칙대로 직전 스냅샷을 STALE 스타일로 유지 표시한다.
7. **금액 가리기 기본값 옵션**: 게이트 해제 직후에도 금액 마스킹 상태로 시작하는 "2단 프라이버시" 설정을 Settings에 추가 — 공공장소 사용자를 위한 보수적 기본값 선택지.
8. **계좌 배분 스택 바의 전역 컴포넌트화(★리뷰 반영 신설)**: 이번 개정으로 Hero의 "통합 자산 구성"이 Asset §3-5와 완전히 동일한 스펙(회색 3단계 교차 + 라벨 직접 표기)이 되었다. 두 문서가 같은 위젯을 각자 따로 정의·유지하는 상태는 여전히 중복이므로, 00 문서에 `AccountAllocationBar` 전역 컴포넌트 계약을 신설해 Asset·Portfolio가 하나의 정의를 상속하도록 정리할 것을 제안한다.
9. **Asset 게이트의 스켈레톤 제거 동기화(★리뷰 반영 신설)**: 본 개정에서 Portfolio 게이트는 60% 스크림 아래의 마스킹 스켈레톤을 제거하고, 대신 해제 애니메이션(페이드+상승)만으로 점프 없는 전환을 구현했다(§1-2). Asset 페이지도 동일한 근거(스크림 위에서 골격이 지각되지 않음)가 그대로 성립하므로, 다음 Asset 문서 개정 시 동일하게 정리해 두 게이트의 시각 언어와 렌더링 복잡도를 다시 맞출 것을 권장한다.
